# 帮助中心改造方案与落地记录

## 背景

当前帮助中心建立于早期版本，内容主体仍停留在 `v3.2.x ~ v4.3.x` 时期。项目现已演进到 `v4.5.25`，在功能范围、设置结构、后台管理与部署更新链路上都发生了明显变化，导致帮助中心与真实产品状态脱节。

本次改造目标不是简单补文案，而是把帮助中心升级为一套可持续维护的“产品文档中心”。

## 目标

1. 让帮助中心覆盖当前版本的主要功能、配置、风控、管理后台与运维能力。
2. 把旧的静态 HTML 文章升级为 Markdown 内容源，降低后续维护成本。
3. 为命令、代码块和整篇文档提供一键复制，提升部署、排障和沟通效率。
4. 让帮助中心支持基于正文的全文搜索，而不是只搜标题和标签。
5. 增加与真实页面的关联跳转，让帮助中心从“说明页”变成“操作入口”。

## 信息架构

本次重构后，帮助中心按 8 个大类组织内容：

- 快速开始
- 核心功能
- 配置中心
- 账号与风控
- 管理后台
- 运维部署
- 故障排查
- 版本演进

每篇文章统一包含以下元数据：

- `id`
- `category`
- `title`
- `summary`
- `tags`
- `audience`
- `relatedRoutes`
- `updatedAt`
- `version`

## 实现方案

### 内容层

- 新增 `web/src/content/help-center/*.md` 作为文档源
- 新增 `web/src/data/help-center.ts` 管理元数据与分类
- 基于 `import.meta.glob` 按文章加载 Markdown，避免一次性把全部正文打进主包
- 保留 `web/src/data/help-articles.ts` 作为兼容出口

### 渲染层

- 使用 Markdown 渲染器统一输出 HTML
- 在帮助中心正文渲染完成后，对 `pre > code` 自动装饰复制工具栏
- 从 Markdown 标题中提取文章目录，生成右侧导航

### 交互层

- 基于现有 `useCopyInteraction` 复用全局复制反馈
- 通过路由 query 持久化当前文章，支持直接分享
- 搜索覆盖标题、摘要、标签、正文与关联路由
- 在设置、账号、流程、日志、公告等页面增加上下文帮助按钮，直接跳转到对应文章
- 在帮助中心文章页提供“过期反馈 / 补充建议 / 本文链接”复制入口，形成文档治理闭环

## 自检与后续建议

本轮改造优先把“帮助中心是否真实反映产品能力”解决掉，同时把复制、搜索和文档维护链路打通。

第二阶段已经把帮助中心从“可读”推进到“可维护”：

1. 新增按角色切换的阅读视角，减少普通用户误读管理员文档。
2. 新增必读文档推荐，把“先看哪几篇”从经验口口相传改成界面内显式推荐。
3. 新增文档时效状态与每篇文档最近改动时间线，帮助判断文章是否新、为什么会这样写。
4. 新增帮助中心与版本更新日志的联动，并通过构建前生成精简索引避免把完整 `CHANGELOG.md` 打进前端 bundle。
5. 新增帮助中心正文懒加载，把文章内容拆成按需加载资源，控制帮助中心主包体积。
6. 在设置、账号、流程、日志、公告等页面新增上下文帮助入口，让帮助从“需要主动搜索”变成“随处可达”。
7. 新增文档治理面板，提供过期反馈、补充建议和文章链接复制能力，形成后续维护闭环。

第三阶段继续把“可维护”推进到“可验证、可追踪”：

1. 补上 Playwright 真实点击回归，覆盖文章切换、正文搜索、命令复制、治理任务复制、上下文帮助跳转和设置页锚点返回。
2. 把设置页和系统更新中心的帮助入口细化到具体分组和锚点，帮助页会保留 `section` 参数并自动滚动到对应段落。
3. 给文档治理补上负责人、最近复核人和待处理状态，让“复制治理任务”不只是模板，而是能直接进跟踪流程的最小工单。
4. 这轮真实回归中发现并修复了一个产品级问题：帮助中心初始化时会覆盖外部传入的 `section` 查询参数，导致从设置页跳入帮助文章时丢失定位锚点。

后续可以继续迭代的方向：

1. 把当前帮助中心回归从单条专用脚本扩展成可复用的 smoke 套件，继续覆盖账号页、公告管理、系统日志等上下文入口。
2. 给帮助中心的“关联页面”与“当前分类帮助”补可观测埋点，统计哪些文档最常被跳转、哪些锚点最常命中。
3. 把文档治理从“复制任务模板”推进到“后台可提单 / 可分配 / 可关闭”的完整工作流。
4. 如果未来帮助中心改成服务端下发，再补 `DOMPurify` 之类的 HTML 安全清洗。
