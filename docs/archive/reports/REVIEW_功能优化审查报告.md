# 系统功能优化深度审查报告 (2026-03-06)

基于最近几日（3月4日 - 3月6日）密集的底层重构与前端优化，目前系统在流畅度、防封性与可扩展性上有了飞跃式提升。为了防止“按下葫芦浮起瓢”，本报告将针对本次涉及的各项大规模变更进行深度审查，评估其潜在风险，并提出后续的优化建议。

---

## 1. 登录架构重构模块 (Login Architecture)

### 主要变更
- 修复了 QQ 扫码 `400` 错误，采用官方入口优先，第三方代理通道回退的自适应机制。
- 重新启用了微信的 iPad / 车机协议扫码入口，屏蔽了支付异常的纯小程序通道。

### 潜在风险与影响
⚠️ **代理降级风控（中风险）**
目前的代理通道 `qqcode.skeo.net` 属于跨区 IP，若官方直连失败而大规模频繁调用第三方代理接口，会导致多个账号的扫码请求指向同一个归属地（如重庆或国外节点），在此刻极易触发腾讯的“异地登录局域网拦截”（即就算扫码了手机端也会被拦截）。
⚠️ **微信协议周期性失效（低风险）**
目前启用的 iPad 与 车机 协议虽然避开了小程序支付不能结算的坑，但此类非常规设备的协议通常在每次微信服务大规模打击外挂时首当其冲被和谐。

### 优化建议 (TODO)
- 后续建议在 `logs/development/Update.log` 中埋入日志上报，定期统计“官方通道”与“代理通道”的成功转化率，若代理通道经常被触发，建议在 `Network.js` 增加提醒用户网络环境不佳的 Toast。
- 可以关注微信官方长效长连接（`Keep-Alive Session`）技术，尽量降低重新扫码的频率。

---

## 2. 日志系统与性能重构框架 (Log Engine)

### 主要变更
- 前端 `Dashboard.vue` 解除了深度拷贝（Deep Copy）和暴力重排序（Sort），直接透传后端的自然序以解决假死卡顿。
- 后端数据源从 `globalLogs` 千条共享池拆分为 `worker.logs` 的独立内存池区，且加入了前端切号强刷屏功能。

### 潜在风险与影响
⚠️ **内存膨胀边界（低风险，但关键）**
为每个子进程 Worker 设立 `1000` 条记录的缓冲池，在小规模部署（几十人~两百人）时完全没有压力。但系统被设计为“大盘服务”级：当单实例挂载 **几千个** 账号发生同时并发且频繁报错时，这几千万条的对象数据会让 Node.js 的 V8 垃圾回收（GC）变得困难，严重时会引起整个 Node 主进程的 OOM 重启。

### 优化建议 (TODO)
- 建议将 `1000` 这个硬编码数字抽出为一个环境变量（如 `MAX_ACCOUNT_LOG_LIMIT`），使用户（比如用小机器部署的人）可以将其降低到 `300` 甚至是 `100` 以节省内存。

## 3. UI与前端架构大一统 (Theme & UI Unification)

### 主要变更
- 修复了 `calculator.html` 和各个农场百科工具在切换暗色模式时周围留白的“边框门”问题。
- 使用 `iframe` 主题劫持方案动态抽取主界面的 Semantic Color 覆盖计算器的默认 Tailwind 类。
- 优化了窄屏情况隐藏侧边栏的兼容结构。

### 潜在风险与影响
⚠️ **FOIT (Flash of Unstyled Text/Theme) 闪烁现象**
因为 `iframe` 必须等 DOM 载入并且主通信完成注入 `<style id="yunong-theme-override">` 后才会实施变色，网络不好的情况下不可避免地会有一瞬间的“原版丑绿色”或“大白底”闪现。

### 优化建议 (TODO)
- 可以在 iframe 的载入页面 `calculator.html` 的顶层 HTML 节点先置入一个 `<div id="loading-mask">` 或者默认为 `#000` 黑底，等接收到父页面 `window.postMessage` 传入的颜色变量渲染完毕后，再撤掉遮罩，做到真正的体验无缝切换。

## 4. CI/CD 与 多架构部署流水线 (Deployment)

### 主要变更
- 实现了 `ARM64` 和 `AMD64` 双向全自动 Docker 构建。
- 重构全栈套件组 `docker-compose.prod.yml` 并修改为 `qq-farm-ui-pro-max` 唯一标识。

### 潜在风险与影响
⚠️ **Action 免费资源额度**
QEMU 模拟构建跨平台架构耗时十分长（部分机器常达 15 ~ 30分钟），会大量占用 Github Actions 的免费计算池长尾。
⚠️ **镜像过大**
如果把包含高频流水日志的 `Update.log` 和一些 SQLite 快照带入到 Image 层缓存中，不仅造成隐私安全风险，而且会导致 Pull 镜像速度变慢。

### 优化建议 (TODO)
- 建议将涉及日志与可变配置的路径（如 `logs/` 或 `store.json`）在 `.dockerignore` 严格排除，转而在宿主机的 `docker-compose.prod.yml` 使用 `volumes: - ./logs:/app/logs` 实现双向动态挂载。

---

> 👨‍💻 **Architect 结语：** 总体而言，系统的重构非常成功。我们在放弃了沉重且昂贵的数据库 IO 后，依然通过内存物理隔离巧妙解决了数据串流与前端阻塞。只需稍加关注极高并发下的“内存刺客”和“限流体验”即可。此系统已达到商用开源可用级别。
